# Sovox — State-of-the-Art Research & Architecture Decisions

**Version 1.0 · July 2026 · Status: Adopted**

This document records the frontier research survey conducted for Sovox and the binding Architecture Decision Records (ADRs) derived from it. Every decision here is grounded either in the Tenzro Network reference implementation (`github.com/tenzro/tenzro-network`, surveyed at HEAD, July 2026) or in the cited external state of the art. Scope: (1) the protocol substrate, (2) decentralized training & inference, (3) confidential & verifiable compute, (4) the agent economy stack, (5) the base operating system, (6) component selections.

---

## 1. The protocol substrate: what Tenzro Network actually provides

Sovox does not reinvent a distributed-computing layer; it operationalizes one. The Tenzro reference implementation is a Rust workspace of 31 crates whose surfaces define Sovox's integration contract:

**Node & roles.** A single `tenzro-node` binary runs one or more roles — `validator`, `model-provider` (model serving, plus compute rental and storage as capabilities on the same stake), `tee-provider`, and `light-client` — under **one stake covering every role**. Renting compute and holding data are not separate `--role` values; they ride the model-provider (`serves_ai()`) capability and settle against the same coverage budget. Compute rental is availability-proof-gated and billed per epoch via streaming escrow; storage is proof-of-retrievability-gated and billed per byte-epoch with Reed–Solomon erasure coding. Provider onboarding (`tenzro join --provider`) performs hardware detection (CPU, RAM, GPUs, TEE), wallet provisioning, faucet funding on testnet, a 100 TNZO compute bond, registration, and pulls the largest catalog model that fits the machine.

**Consensus.** HotStuff-2 BFT with reputation-weighted proposer election, stake-weighted quorum with a 10% per-validator cap, equivocation slashing, and hybrid Ed25519 + ML-DSA-65 post-quantum signatures on safety-critical messages. TEE-attested validators receive a 1.5× leader-selection multiplier — confidential compute is a consensus primitive, not middleware.

**Identity.** TDIP (`did:tenzro`) with four identity classes — humans, delegated agents, autonomous agents, and institutions (LEI-anchored) — KYC tiers, delegation scopes, cascading revocation, W3C DID/VC compatibility, ERC-8004 mirroring, and GDPR Art. 17 erasure. Wallets are FROST-Ed25519 (RFC 9591) 2-of-3 threshold with an ML-DSA-65 hybrid PQ leg.

**Networking.** Dual-plane: libp2p control plane (gossipsub, Kademlia, AutoNAT v2, Circuit Relay v2, DCUtR for permissionless NAT traversal) and an iroh content-addressed data plane (QUIC, iroh-blobs, `tenzro://` URIs, Pkarr discovery byte-bound to the TDIP key). One structural port: **9000 TCP+UDP**. RPC (8545), Web API (8080), MCP (3001), A2A (3002) are meant to be loopback-bound and fronted by a TLS proxy.

**AI surfaces.** Multi-modal serving through llama.cpp (CUDA backend) for LLMs plus ONNX Runtime execution providers for forecast/vision/embedding/segmentation/detection/ASR modalities; a price/latency/reputation inference router; MTP speculative decoding; license-tier gating. **Mixture-of-Experts serving is a first-class, live-network-verified path**, in two modes over one provider population: *full-replica* when a single provider's hardware fits the model, and *decentralized expert-shard* when it does not — providers declare the experts they hold (`ProviderCapacity.moe_holdings`), a dispatch planner aggregates per-token top-k routing into per-holder batches over each holder's iroh QUIC endpoint, and the full model is assembled across distributed memory that **no single node holds**. Typed pipeline roles (Replica/Router/ExpertHolder/Prefill/Decode), two-tier expert residency (memory LRU + disk spill, auto-budgeted), Q8_0 activation compression, and block-quantized experts (Q4_K_M default) harden the fan-out; `tenzro-cluster` additionally splits a single provider's model across LAN boxes via layer-wise pipeline parallelism, so several commodity machines jointly serve a model none of them fits alone. **Tenzro Train** implements a Decoupled-DiLoCo-style protocol: Nesterov outer optimizer, Int8/Int4 blockwise gradient quantization, signed per-gradient receipts, on-chain run-root commitments, and sponsor-selected trust tiers (Open / Verified / Confidential). Phase 1 ships single-modality (timeseries) training with **simple mean aggregation**; Byzantine-robust aggregation (TrimmedMean, coordinate-median, Krum) plus redundant-assignment slashing are the **Phase 2 roadmap**.

**Verification & TEE.** Plonky3 STARKs over the KoalaBear field (transparent setup) with prebuilt AIRs for inference/settlement/identity, anchored by `ZK_VERIFY`; hardware attestation for Intel TDX, AMD SEV-SNP, AWS Nitro, and NVIDIA GPU Confidential Computing verified on-chain by `TEE_VERIFY` (Intel Tiber Trust Authority chain validation included).

**Payments & agents.** x402 v1, AP2 v0.2 mandates, ERC-8004 v0.6+ trustless-agent registries, ERC-4337 v0.8, Stripe SPT, Tempo, Visa TAP and Mastercard Agent Pay adapters; 490+ JSON-RPC methods, 416 MCP tools, 42 A2A skills exposed by the node; a WASI 0.2 component host (`tenzro-wasm`) for sandboxed, fuel-metered agent skills. TNZO is fixed-supply (1B), the protocol denominator for gas, bonds, commissions, and governance.

**Deployment assumptions (what the OS must satisfy).** The upstream operator guide is deliberately OS-agnostic ("the container is the abstraction"), recommends 4 vCPU / 16 GB RAM / 100 GB SSD per validator, ≥4 validators for BFT (10+ production), TCP keepalive sysctls for cross-region NAT survival, digest-pinned images, glibc 2.31+, Rust 1.85+ toolchain, CUDA 12.6 + cuDNN for the GPU build (compiled for SM 75–90), ROCm 6+ and Vulkan/Metal alternatives, and warns that the default RPC bind is 0.0.0.0 and must be firewalled or proxied. **Sovox exists to make every one of these lines a declarative default instead of a runbook step.**

## 2. Decentralized training — state of the art (mid-2026)

The low-communication training lineage that Tenzro Train implements is now the proven mainline of decentralized training:

- **DiLoCo → OpenDiLoCo → INTELLECT-1** established feasibility: a 10B model trained on 1T tokens across three continents, with DiLoCo + custom int8 all-reduce achieving up to ~400× communication reduction versus naive data parallelism (Prime Intellect, Dec 2024).
- **Streaming DiLoCo** (DeepMind, 2025) overlaps synchronization with compute; combined with 4-bit quantization it cuts bandwidth strain ~100× versus naive DP (Epoch AI analysis, Dec 2025).
- **Scale keeps climbing:** SparseLoCo-style quantization+sparsification underpins Templar's 72B decentralized run and Covenant AI's Covenant72B; DiLoCoX (China Mobile/0G) demonstrated a 107B foundation model over ~1 Gbps links; Pluralis' Agora explores pipeline-parallel WAN training where no contributor holds the full model.
- **Decoupled DiLoCo** was announced by Google DeepMind (Apr 23, 2026), pairing Pathways-style asynchronous orchestration of heterogeneous accelerators with DiLoCo's bandwidth minimization for multi-datacenter frontier training.

**Implication for Sovox.** Tenzro Train's Decoupled-DiLoCo design is aligned with the frontier trajectory, not a niche fork. The OS contribution is operational: GPU-node images with pinned CUDA/ROCm userspace, deterministic driver/toolkit pairing, isolation of untrusted training payloads (§ADR-006), bandwidth shaping so training sync never starves consensus traffic, and checkpoint/gradient storage on snapshot-capable filesystems. Byzantine-robust aggregation (TrimmedMean/coordinate-median/Krum) is specified in `tenzro-training` as the **Phase 2 hardening step** — Phase 1 ships simple mean aggregation with stake bonding and the Open trust tier — matching the academic consensus that open-participation training needs robust aggregation plus stake/slashing. Sovox surfaces the sponsor-selected trust tier (Open / Verified / Confidential) as a one-line role option and tracks the aggregation-rule rollout as an upstream dependency.

## 3. Distributed & decentralized inference — state of the art

Centralized serving SOTA (vLLM/SGLang continuous batching, disaggregated prefill–decode à la NVIDIA Dynamo, Kubernetes-native llm-d) optimizes within a datacenter; decentralized serving (Petals-style swarms, exo-style local clusters) optimizes across owners. Tenzro's design sits deliberately between: each provider serves whole models (or declared MoE expert shards) on hardware it owns, and the *network* performs routing by price/latency/reputation with per-call settlement and optional proof. This sidesteps the latency cliff of layer-sharding across the WAN while preserving open participation, and it matches where the 2026 market went: sovereign buyers want *whole-model, attested, locally-hosted* inference, not anonymous layer soup.

**Implication for Sovox.** Default serving path is the native `tenzro-node` engine (llama.cpp CUDA + ONNX Runtime EPs) because it is what the protocol meters, prices, and proves today. Sovox additionally defines a **high-throughput serving profile** (vLLM or SGLang behind the node's OpenAI-compatible surface) as an optional module for large-VRAM operators, tracked as an upstream engine-adapter integration (§ADR-007). `tenzro-cluster`'s deterministic HRW placement handles LAN-scale multi-box serving without a coordinator.

## 4. Confidential & verifiable compute — state of the art

- **GPU TEEs are production-real.** NVIDIA Confidential Computing is supported on H100/H200 (Hopper) with ~2–5% published overhead for typical LLM inference and GPU-specific attestation from the GPU security processor; **Blackwell is the industry's first TEE-I/O-capable GPU**, adding inline NVLink protection at near-parity throughput, and the capability carries into Rubin. End-to-end CPU+GPU attestation flows (Intel TDX + NVIDIA CC verified via Intel Trust Authority / NVIDIA Remote Attestation Service) are shipping on HGX B200 systems. AMD MI300X-class parts sit *inside* the host SEV-SNP envelope (VM-level, not GPU-level attestation) — a distinction regulated buyers care about.
- **CPU TEEs:** AMD SEV-SNP (EPYC Milan+) and Intel TDX (4th-gen Xeon+) are mature guest targets in kernel 6.12 LTS; AWS Nitro covers the cloud-enclave case. Sovox targets SNP and TDX as first-class *host* capabilities for the TEE-provider role.
- **Verification spectrum economics.** zkML remains orders of magnitude too expensive for frontier-scale models; TEE attestation is near-free at runtime; optimistic/replication schemes sit between. Tenzro's hybrid — STARK proofs where circuits are small (settlement, identity, bounded inference AIRs) and hardware attestation where they are not, both anchored on-chain — matches the 2026 consensus on practical verifiable AI. Sovox's job is to make attestation *boring*: measured boot to TPM 2.0 PCRs, CVM launch with SNP/TDX reports, GPU CC evidence, all flowing into `TEE_VERIFY` without operator ceremony.

## 5. The agent economy stack — state of the art

- **Governance converged.** MCP, A2A, and the merged ACP all sit under Linux Foundation stewardship; the Agentic AI Foundation (Dec 2025; Anthropic, OpenAI, Google, Microsoft, AWS, Block among co-founders) hosts the layer. MCP reached ~97M monthly SDK downloads by late 2025; A2A passed 150+ supporting organizations with v1.0, signed Agent Cards, and GA integrations across the major clouds by April 2026. The two-layer default — MCP for agent↔tool, A2A for agent↔agent — is exactly the surface `tenzro-node` already exposes (416 MCP tools, 42 A2A skills).
- **Payments converged too.** The x402 protocol moved to the Linux Foundation's x402 Foundation (April 2026; Coinbase contribution; AWS, Cloudflare, Anthropic, Circle, Amex, Mastercard, Stripe, Visa, Shopify among participants). Coinbase reported ~169M payments across ~590K buyers and ~100K sellers in year one; AWS CloudFront/WAF x402 support went GA June 15, 2026 and Cloudflare opened its Monetization Gateway. Honest caveat, carried into our whitepaper risk section: independent on-chain analysis attributes a large share of headline volume to self-dealing/wash activity, and real daily settled volume remains modest — agentic payments are *infrastructure-real but demand-early*. AP2 mandates count 60+ payment/finserv organizations. ERC-8004 trustless-agent registries round out identity/reputation.
- **Implication for Sovox.** The Agent Hub role is a packaging exercise, not an R&D bet: Tenzro already implements x402 v1 (four schemes), AP2 v0.2, ERC-8004, TAP/Agent Pay, plus the WASI 0.2 skill sandbox. Sovox contributes hardened defaults (loopback-bound MCP/A2A behind Caddy with TDIP-scoped auth), a local agent dev environment on Desktop, and per-skill fuel/spend budgets surfaced in `sovox.toml`.

## 6. Sovereignty & market context (for design, not just GTM)

Global spending on sovereign AI systems is projected to exceed **$100B in 2026**; category sizing puts sovereign AI infrastructure at ~$61B (2025) → ~$79B (2026) on a ~28% CAGR toward ~$727B by 2035, with 47 national programs active and platform/orchestration services the fastest-growing layer. Europe's levers (SecNumCloud, BSI IT-Grundschutz, EUCS) make *attestable locality* a procurement requirement, and analysts converge on a **hybrid sovereignty** model — selective control over compute, data, and identity rather than autarky. That is precisely the layer Sovox occupies: sovereignty enforced at the OS and protocol level on hardware the operator owns. Full figures and citations in `04-GTM.md` §1 and References.

## 7. Base operating system evaluation (the "is NixOS right?" question)

The original vision named NixOS. We treated that as a hypothesis, not an answer, and scored the credible 2026 alternatives against Sovox's actual requirements.

**Evaluation criteria** (weighted for a sovereign multi-role AI node): R1 whole-system reproducibility & auditability · R2 declarative multi-role composition (one config tree drives validator/AI/web/mail/agent variants and a Desktop edition) · R3 atomic update + instant rollback · R4 NVIDIA/CUDA + ROCm enablement · R5 TEE host/guest stack (SNP/TDX, kernel & VMM currency) · R6 package breadth (Rust 1.85 toolchain, llama.cpp, ONNX, ZFS, Caddy, Stalwart, Cockpit, KDE) · R7 fleet provisioning & bare-metal install story · R8 sovereignty (no mandatory vendor store/registry/subscription; fully self-mirrorable) · R9 community velocity & hiring pool · R10 low-touch operator UX achievable on top.

| Candidate | Verdict | Notes against criteria |
|---|---|---|
| **NixOS 26.05 "Yarara"** | **Selected** | Only candidate where the *entire* system — kernel, drivers, services, users, firewall — is one evaluable, diffable, rollback-able expression (R1–R3 native; generations + new default systemd stage-1 initrd). Nixpkgs breadth covers every Sovox component incl. CUDA/ROCm packaging and KDE Plasma (R4, R6). `nixos-anywhere` + `disko` give one-command bare-metal provisioning (R7). Zero vendor tether; binary caches are self-hostable (R8). Weaknesses (learning curve, secrets, eval cost, no mature SELinux) are mitigated below rather than disqualifying. |
| Fedora/CentOS bootc ("image mode") | Strong runner-up; pattern adopted, base rejected | bootc's atomic OCI-image OS model matured fast (RHEL 10 image mode, download-only staging Mar 2026, Fedora base-image initiative). But Containerfile `RUN` layering is imperative — reproducibility is per-image-digest, not per-derivation; multi-role *composition* means an image matrix, not a module system; Red Hat gravity conflicts with R8. **We adopt its delivery ergonomics (signed appliance images, three-command update UX) on top of Nix (ADR-002).** |
| Talos Linux | Rejected as base; design inspiration | Superb minimal API-managed appliance — but Kubernetes-only by construction. Sovox nodes are multi-role sovereign servers (mail, web, agents, TEE VMs), not k8s workers. We borrow its no-shell-by-default posture for Server hardened profile. |
| Ubuntu Core | Rejected | Snap confinement is real, but store-centricity, kernel/driver snap friction for fast-moving CUDA stacks, and proprietary store infrastructure fail R8/R4. |
| openSUSE MicroOS / Leap Micro | Rejected | Solid transactional-update host; weaker on R2 (no comparable module system), smaller AI packaging surface. |
| Flatcar Container Linux | Rejected | Excellent A/B container host; general-purpose role breadth (mail/desktop/TEE VMM variety) out of scope. |
| GNU Guix System | Rejected | Matches Nix on R1 philosophically; fails R4 pragmatically (libre-only defaults make NVIDIA/CUDA an unsupported bolt-on) and R6/R9 on ecosystem size. |
| Debian/Ubuntu + Ansible/IaC (status quo) | Rejected as product basis | This is what Tenzro's own deploy guide assumes today; it's exactly the drift-prone, snowflake-prone baseline Sovox exists to replace. Kept as the compatibility target (Tenzro binaries must keep running there). |

**Decision: NixOS stands — now validated rather than assumed — with four mitigations that define Sovox's engineering identity:**

1. **Intent layer over Nix** — operators write `sovox.toml` (or click the wizard); `sovoxd` compiles it to pinned Nix modules. Nix is the substrate, never the required interface (ADR-004).
2. **Image-based delivery** — releases ship as signed, reproducible disk/installer images built from the flake; updates are closure downloads with health-gated activation and boot-counter auto-rollback. bootc-class UX, Nix-grade determinism (ADR-002).
3. **Security hardening stack** — Lanzaboote UEFI Secure Boot, TPM 2.0 measured boot + `systemd-cryptenroll` disk unlock, impermanent root on ZFS, systemd unit sandboxing, and microVM/CVM isolation classes standing in for the missing SELinux story (ADR-005/006).
4. **Sovereign supply chain** — every input flake-pinned; Sovox binary cache signed (Ed25519 narinfo) and fully mirrorable on-prem; SBOM (SPDX) and build provenance per release; no telemetry without opt-in (ADR-009).

## 9. Risks & open questions (tracked)

1. **Upstream engine adapters (ADR-007)** — vLLM/SGLang metering parity with the native engine needs protocol-side receipts; coordinate with Tenzro before v0.2.
2. **GPU CC passthrough breadth** — SNP/TDX CVM + NVIDIA CC is validated on Hopper-class server parts; consumer-GPU TEE coverage is absent industry-wide. TEE-provider role hardware matrix stays conservative.
3. **NixOS CUDA cache economics** — CUDA closures are large; Sovox must operate a well-funded binary cache (and jumbo image variants) so no operator ever compiles CUDA.
4. **Agentic-payment demand reality** — x402 rails are deployed at hyperscaler edge, but organic demand is early (see honest-caveat data in §5); revenue projections in GTM use conservative utilization.
5. **Testnet→mainnet economics** — bonds, rewards, and slashing parameters cited from Tenzro docs reflect the current testnet-era spec; Sovox surfaces them as data, not constants.
6. **Selinux-class MAC** — deferred in favor of VM-boundary isolation (ADR-006); revisit if a NixOS SELinux/landlock story matures.
